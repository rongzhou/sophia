# Installation and Build

Sophia is a Rust workspace (Cargo workspace). This document explains how to build, test, and run it from source.

## Prerequisites

| Dependency | Version / Notes |
| --- | --- |
| Rust toolchain | `rustc` >= **1.95** (edition 2021), including `cargo`. Installing with [rustup](https://rustup.rs/) is recommended. The project follows the latest stable release, and CI guards this MSRV (see `rust-version` in `Cargo.toml`). |
| C compiler | Required to compile the vendored Tree-sitter parser (`core/syntax/src/parser.c`). Use `gcc`/`clang` on Linux, or Xcode Command Line Tools on macOS. |
| `python3` (optional) | Required **only** by the benchmark `baseline` mode, which executes Python candidates generated by the LLM. Normal build / test / e2e does not need it; when absent, benchmarks automatically run only `sophia` mode. As a runtime external tool, it does **not** enter the Cargo dependency tree. |

Notes:

- **SQLite does not need to be installed system-wide** — `workflow/graph-db` uses the `bundled` feature of `rusqlite`, so SQLite is built with the project.
- **Normal builds do not need the Tree-sitter CLI** — the grammar has already been generated into the vendored `parser.c` (ABI 15), and `build.rs` only compiles it. `tree-sitter-cli` 0.26.x is needed only when **modifying `grammar.js`** and regenerating the parser (see “Changing Syntax” below).
- Network: core builds need only crates.io. Network access and API keys are needed only when running workflow commands / e2e tests that call LLMs.

## Build

```bash
# After cloning, from the repository root
cargo build --workspace
```

Optimized release build:

```bash
cargo build --workspace --release
# Binary is at target/release/sophia
```

## Test

```bash
# Whole-workspace unit / integration tests (deterministic; no LLM calls)
cargo test --workspace
```

Code style and static checks (matching CI):

```bash
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
```

## Install the `sophia` Command

```bash
# Install CLI from the workspace to ~/.cargo/bin
cargo install --path cli

# Then use it directly
sophia --help
```

Or run it directly through Cargo without installing:

```bash
cargo run -p sophia-cli -- --help
```

## End-to-End (Real LLM) Tests (Optional)

e2e tests make real network requests and consume nondeterministic LLM outputs, so they are `example`s and do **not** enter the `cargo test` gate.

```bash
export SOPHIA_LLM_API_KEY=<your-key>     # Required for OpenAI-compatible mode; not written to disk / graph / logs
cargo run -p sophia-cli --example e2e -- --list          # List all cases (no key needed)
cargo run -p sophia-cli --example e2e -- --case G1-01    # Run one case
cargo run -p sophia-cli --example e2e -- --group g1      # Run one group
cargo run -p sophia-cli --example e2e -- --llm-mode ollama --case G1-01
```

Optional environment variables: `SOPHIA_LLM_MODE`, `SOPHIA_LLM_MODEL`, `SOPHIA_LLM_BASE_URL`, `SOPHIA_LLM_MAX_REPAIRS`.
OpenAI-compatible mode skips cleanly when `SOPHIA_LLM_API_KEY` is not set; Ollama mode defaults to local
`http://localhost:11434` and model `qwen3.6:latest`, with no API key required. For serial batch execution, see `scripts/run_e2e.sh`.
See `docs/en/e2e_test.md`.

## Programming-Capability Benchmarks (Optional)

These compare “LLM writes Python directly” vs “Sophia workflow” on small tasks by **success rate + elapsed time**. They are also `example`s and do **not** enter the `cargo test` gate.

```bash
export SOPHIA_LLM_API_KEY=<your-key>     # Required for OpenAI-compatible mode
cargo run -p sophia-cli --example benchmark -- --list            # List tasks (no key needed)
cargo run -p sophia-cli --example benchmark -- --task abs_difference   # Run one task (both modes)
cargo run -p sophia-cli --example benchmark -- --level l1         # Run one level
cargo run -p sophia-cli --example benchmark -- --mode sophia      # Run only one mode
cargo run -p sophia-cli --example benchmark -- --llm-mode ollama --mode sophia
```

`baseline` mode executes Python generated by the LLM and requires `python3` (when absent, only `sophia` mode runs automatically). Outputs are written to `sophia-runs/benchmark/<label>/{runs.jsonl,summary.md}` (the two core metrics are success rate / elapsed time). For serial batch execution, see `scripts/run_benchmark.sh`. See `docs/en/benchmark_test.md`.

> Credential management: you can put keys in `.secrets/llm.env` (ignored by `.gitignore`) and load them with `set -a; source .secrets/llm.env; set +a`, avoiding plaintext in shell history.

## Changing Syntax (Contributors Only)

If you modify `core/syntax/grammar.js`, regenerate the parser with the aligned Tree-sitter CLI version:

```bash
# Requires tree-sitter-cli 0.26.x (aligned with tree-sitter crate 0.26 + ABI 15)
cd core/syntax
tree-sitter generate --abi 15
```

Version alignment is a hard constraint (see `docs/en/engineering_notes.md`): the tree-sitter crate, CLI, and generated `parser.c` ABI must match.

## Troubleshooting

- **Linker / C compiler errors**: confirm that a C compiler is installed (`cc --version` or `gcc --version`).
- **`rustc` version too old**: run `rustup update` to upgrade to >= 1.80.
- **First build is slow**: the `bundled` feature of `rusqlite` compiles SQLite; this is a normal one-time cost.
