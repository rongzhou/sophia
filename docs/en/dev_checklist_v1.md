## I. Overview

Phase goal (v1): turn Sophia from a “prototype with only interpreter” into a “serious language that can compile and be deployed” (`language_design.md` §1.1 Goal 1). Two parallel workstreams, both required:

- Workstream A — WASM codegen: extend the execution backend from interpreter to deployable WASM artifact. The interpreter is not retired; it becomes the equivalence oracle / differential-testing baseline. Extend strip-assist equivalence to byte-level artifact diff.
- Workstream B — Language and standard library expansion (demand-driven): not a fixed feature list; instead minimal expansions gated by design and triggered by concrete demo demands. v1 scope is capped by three demo demands (§II “v1 Demo Requirements”): D1 failure modeling, D2 network fetch + intent safety (flagship LLM-native demo), D3 a serious pipeline. This implies the minimal expansion set F1 (unified type syntax + fallible returns via `one of {...}`, Completed) + F2 (Http effect family) + S1 (HTTP client host in stdlib) + S2 (stdlib prompt scaffolding). Items outside the starter subset such as `entity.with` / cross-domain data flow / contract proofs / `task` execution are explicitly deferred to v2+.

v1 completion criteria (all three required):
1. Starter-subset programs compile via WASM backend and are equivalent to the interpreter on each hidden case (differential tests all green).
2. All three v1 demo requirements (D1 fallible returns / D2 Http+intent / D3 serious pipeline, §II) run end-to-end in sophia mode (benchmark level L6+). Among them, D2 must produce one real accept/reject matrix entry (Sophia statically rejects an unsafe candidate that TS baseline would accept).
3. Strip-assist equivalence holds at the artifact layer (byte-for-byte equality).

Note: Criterion 2 is bounded: v1 language expansion is capped by D1/D2/D3 (i.e., F1 + F2 + S1 + S2). No expansion lacking a triggering demand.

Starting point: v0 core chain (parse → HIR → semantic → exec-ir → interpreter run) + workflow graph loop + 6 e2e groups + benchmark ladder L1–L5 already run with real LLMs; workspace 299 passed / 0 failed. v1 coding not yet started at the initial point.

## II. Worklist

### v1 Demo Requirements (demand-driven; cap for Workstream B)

Methodology: Demand → expansion, not expansion → find a use. Two admission channels only: (1) Demo-driven — a compelling demo/benchmark needs a capability, add the minimal expansion needed and cap by the demand. (2) Strongly-argued LLM-native feature — even higher bar, requires a clear argument for why it specifically serves autonomous LLM programming plus an accept/reject or measurable benefit. Anything that passes neither goes to v2+ and is out of v1 scope.

| Demand | What to show | Why compelling | Triggers |
| --- | --- | --- | --- |
| D1 Failure modeling | Computations/storage flows that may fail are expressed as explicit mutually exclusive outcomes via `one of { T, SomeError }`; caller must match all members | v0 only had `raise` (non-recoverable) and “nullable but missing reason”; “recoverable with reason” is a real gap; `one of` expresses members directly with no wrapper boilerplate | F1 ✅ |
| D2 Network fetch + intent safety (flagship) | Action fetches untrusted data via `Http.Get` (type is `Raw<Text>`); downstream consumption must go through an explicit intent conversion or be statically rejected; contrast TS+tsc that accepts “use fetched string as trusted value directly” | Lands the flagship thesis (language makes “what data went through” machine-checkable), yields a real accept/reject matrix. Reuses already-built intent + capability/effect + host mechanisms; small expansion, large persuasion | F2 + S1 + S2 |
| D3 Serious pipeline | 3–5 actions cooperating end-to-end with a fallible step and an effect boundary (fetch → validate → compute → persist) | Verifies that, after expansion, the language can express programs clearly beyond v0; no new features, only integrates F1 + existing capabilities | None (integration acceptance) |

Not in v1 (defer to v2+; no v1 demo trigger): cross-domain/library collaboration demo (needs boundary + `sophia.lock`, report S3), multi-round evolution demo (needs edit transition/Evolution Boundary, report S2). Valuable but large scope, independent subsystems; would blow up v1’s boundary.

### Workstream B — Language and Stdlib expansion (demand-driven + design gates)

Discipline: B is not a fixed implementation sequence; it is demand-driven (§v1 Demo Requirements). Each expansion is triggered by a specific demo and must pass a design gate (standalone design doc → discussion and acceptance → then full-stack implementation + regression tests + docs), same paradigm as `effect` and storage. Anything that does not meet either of the two admission channels goes to v2+ and is out of v1.

Syntax principles (`language_design.md` §3): do not mimic generics/templates/macros/traits; semantics first, no omissions, do not fear verbosity. `<>` is exclusive to Intent wrapper. Structural types use the `of` keyword family (`list of`/`one of`/`schema of`), all are closed built-ins (not a generic system). Fallible returns via `one of` are constructed directly and matched directly; no `?`/`unwrap` sugar.

Stdlib scope principles (demand-driven, functional libraries): the standard library is a functional library and grows incrementally per demo. Only provide functional layers needed; do not build protocol stacks (e.g., D2 only needs the Http.Get function, so only provide that; do not build TCP/IP, TLS, sockets — leave that to the host/runtime’s mature implementations). No demand, no addition.

Each feature proceeds in three phases: design gate → layered implementation → demo acceptance.

#### F1 — Unified type syntax (`one of` / `list of` / `<>`-exclusive intent) + fallible returns (from D1/D3) ✅

Admission: create a checkable invariant “failure branches must be handled explicitly” and reduce LLM missing-failure-path issues. The initial Rust-like `Result<T,E>` was rejected (Rust mental model bias; wrapper carries no information). Adopt `one of {...}` union type to express fallible/nullable returns directly (members are themselves, no wrapper), and in the process unify the entire type syntax. Final design in `docs/type_system.md` (replaces deleted `result_type.md`).

Core rule (no exceptions): `<>` is for Intent type only; structural types use the `of` family (`list of T` / `one of { M, ... }` / `schema of T`). Remove `Optional<T>`/`List<T>`/`Schema<T>`/`Some`/`None`/`<optional>.exists`, add built-in `Null`, and introduce type patterns in match.
